# Skill · 导演式 VLM 审片（Director VLM Review）

> **何时读我**：每个视觉产物落盘后、调 `vlm_understand` 之前。
> **关键工具**：`vlm_understand` / `extract_frames`
> **核心：VLM 是裁判，不是法官**。让它判断"能不能进下一步"、"哪里能改"，不要逼到完美。

|---|---|---|---|

| 4 | **把"过审/不过审"格式约束放在 question 开头/第一句** ⭐ | `整体结论：过审/不过审。我要的是明亮的现代简约客厅...` | 这是**输出格式约束**，逻辑位置应该在 **question 末尾**（开头先身份代入 + 导演意图，末尾才接格式约束）。塞开头 → VLM 把整段当成"是非题"应付 → 谄媚 100% 命中 |

任一条命中 → **不许调 vlm_understand**，重写 question 后再来。

### question 必须是三段式结构

```
[ 第一段：身份代入 + 本片诉求 ]      ← 让 VLM 切到导演视角
[ 第二段：导演意图正向描述         ]      ← "我要的样子是 XXX"，不列清单不写问句
[ 第三段：输出格式约束             ]      ← "请独立观察... 不要被上面带偏... 第一行：整体结论：过审/不过审..."
```

**死罪 4 的反例就是把第三段直接放最前面**——格式约束写在哪里只是技术问题，但 VLM 是按提问顺序处理 token 的，第一句话定调整段提问的框架。

### 怎么写才对（30 秒速记版）

写 question 时**核心思维转换**：从"我要 X，你看是不是 X"的对答案题 → "**整片 context + 导演该看的维度**，你自己评判"的发散题。

具体到代码层面只做一件事：**第二段写"作为这部 XX 类型片的导演，你会从哪些维度去看这个产物"，不写"我要 A+B+C+D 清单"**。完整 question 模板按产物类型分**场景 A（entity 基准图）/ 场景 B（单 shot 视频）/ 场景 C（全片成片）**三种，详见文末「✅ 正例」节——**整段直接抄**，把 `<尖括号>` 部分换成你当前项目的内容。

```
✅ 正例（场景 B 单 shot 视频，三段式精简版）：
   第一段：「你是 30 秒校园霸凌警示公益片的总导演。本片故事线：小女孩误把
            垃圾扔进回收桶，老奶奶温柔提醒...。本片调性：暖色温情。
            现在你正在审第 3/4 镜头：承上启下表情特写...」
   第二段：「作为这部片的导演，你需要从下面几个维度评判这一镜：
            1. 本镜叙事意图传达到位了吗
            2. 演员的表演（表情/微动作/眼神）是不是演到了
            3. 画面质感跟整片调性是不是同一个片子
            4. 跟前后镜头能不能接上
            ...挑你真正在意的点说。」
   第三段：「请独立观察这段视频实际拍出来什么样，不要被上面带偏。
            第一行：整体结论：过审 / 不过审；
            过审：说出你具体看上哪些点；
            不过审：说出导演视角的问题诊断+你想要的样子（不需要写 prompt
            格式，工程化翻译是 agent 走 PE 7 步该做的事）。」
```

⚠️ **关键分工**：VLM 是导演（说"问题在哪 / 想要什么"），**不是 prompt 工程师**。让 VLM 直接出成品 prompt 必踩工程坑：

| VLM 不会做的事 | 后果 |
|---|---|
| 4 类特殊符号（`<音效>` `{台词}` 禁 `（）BGM`） | 写"老奶奶夸奖"不补 `{真棒！}` → 张嘴无声 |
| `reference_entities` 哪些 entity 该传 | 只用文字描述硬凑 → 一致性崩 |
| 时长比例双保险 | 不写 → Seedance 出错画幅 |
| BGM lint（`gen_video_t2v` 不许 `（BGM）`）| 写"配乐"括号 → 直接 raise |

**正确分工**：VLM 出导演视角的"问题诊断+想要的样子" → agent 走 PE 7 步翻译成符合规范的 prompt → 喂回工具。

```
❌ 反例（来自实际项目 reviews/）：
   「你是本片导演，正在审核小区垃圾桶区域场景的宽景基准图。我要的样子是
    小区楼下露天垃圾桶摆放区，周围有绿色灌木，地面铺浅灰色地砖，干净
    整洁，下午暖黄色阳光斜照，氛围温暖。请独立观察图片内容，不要被
    上面的描述带偏。第一行：整体结论：过审/不过审...」
   → 1) 缺整片 context（这是 30s 公益小视频还是 MV？调性？观众？）
     2) "我要的样子是 A+B+C+D" 是清单不是发散维度
     3) VLM 只会照清单复读"是的我看到了灌木+地砖+暖阳光"，发现不了
        "虽然元素都对，但场景跟整片其他镜头风格不统一"这种导演级问题
```

## 铁律：每个产物都看一眼

任何视觉产物落盘后立刻 `vlm_understand` 检查一次，**不允许跳过**：

- `entity_add_view` 落盘 → vlm（**基准图必须 vlm 过审才能继续做其他视角；其他视角和基准图各审一次，character 三视图全部要过**）
- `gen_image` 落盘 → vlm
- `query_video_task` 拿到 mp4 → vlm（这是单镜头的"出完时刻"，**不是 `gen_video_t2v` 提交那一刻**）
- 链式生成时，**上一段视频不过 vlm 不许出下一段**
- 合成成片落盘 → vlm

### 自检：开拍前先点数

`gen_video_t2v` 之前**必须自检**：扫一眼 `entities/<X>/*.png` 和 `reviews/*.json`：

- `entities/` 下有几张 png（不算 ref.json）
- `reviews/` 下有几条 review 是审 entity 视图的

**两个数字必须相等**。少一张就回去补 `vlm_understand`，不许"先开拍再补审"。
（推荐用 `code_run` 跑一句 `ls projects/<pid>/entities/*/*.png | wc -l` 和 `ls projects/<pid>/reviews/review_*.json | wc -l` 自己核一遍）

### 命名规范（必须遵守）

调 `vlm_understand` 时 **必须传 `name` 参数**，按场景规范命名，否则 reviews/ 目录会变成一堆 `understand_<时间戳>.json` 的天书：

| 场景 | name 格式 | 例子 |
|---|---|---|
| 审 entity 视图 | `review_entity_<entity>_<view>` | `review_entity_dancer_girl_front` |
| entity 视图返工 | `review_entity_<entity>_<view>_v<K>` | `review_entity_little_boy_shooting_v3` |
| 审单镜头视频 | `review_shot_<shot_id>` | `review_shot_s01` |
| 单镜头返工 | `review_shot_<shot_id>_take<K>` | `review_shot_s03_take2` |
| 审合成成片 | `review_compose_v<版本号>` | `review_compose_v1` / `review_compose_v2` |
| 非审片（描述/对比/定位） | `understand_<目的>` | `understand_compare_s01_v1_v2` |

**禁止**用默认值 `understand`、单纯时间戳、`s01_review`、`check`、`review_001` 这种乱序/无对象命名。文件名要让你三个月后回来翻还能一眼看懂审的是哪段。

### 不过审的处置（hard stop）⭐

当 `review_shot_<shot_id>` 的 answer 第一行含「**不过审**」/「不符合」/「需修改」/「需重做」：

1. **绝对禁止**对该 shot 执行 `video_concat` / `video_crossfade` / `audio_amix`
2. **绝对禁止** `start_long_term_update` 收尾交付
3. **必须**按 review answer 列出的"修改方向"原文当 prompt 重做该 shot（见上方第 4 条硬约束）
4. 重做后用 `_take<K>` 后缀重新审（`review_shot_s03_take2`）
5. 同一 shot 最多返工 2 轮，仍不过则标记 best_effort 并 `ask_user`，不许偷偷拼接

> 已踩坑 · p20260529_170607_：shot_s03 review 写明"不过审 + 5 个具体问题"，agent 直接 `video_concat` 把不过审的 s03 拼进成片然后 `start_long_term_update` 收尾。这是字面违反"分镜 = 交付契约"。

## 提问模板核心原则（每次都带这四件事）

1. **整片 context**（"这是什么类型片，调性是什么，目标观众是谁，从 `storyboard.synopsis` 取"）—— 让 VLM 进入项目语境
2. **当前产物在流水线的位置**（"这是 entity 基准图 / 第 X/N 镜头 / 全片合成成片"）—— 让 VLM 知道审什么粒度
3. **导演该发散思考的维度**（"作为本片导演你会从哪些角度看这个产物"）—— 给思考框架，**不给"我要 A+B+C"清单**
4. **不过审时要"导演视角的问题诊断+想要的样子"**（**不是工程化 prompt**）—— agent 拿到诊断走 PE 7 步翻译成符合规范的 prompt 再喂回工具

> 完整 question 模板按产物类型分场景 A / B / C 直接抄，详见本文末尾「✅ 正例」节。
>
> ⚠️ **职责分工铁律**：VLM = 导演（出"问题诊断+想要的样子"），agent = PE 工程师（走 [skill_prompt_engineering.md](../skill_prompt_engineering/skill_prompt_engineering.md) 7 步把诊断翻译成合规 prompt）。两边职责不能混——VLM 直接出 prompt 必踩 4 类符号 / 5 大死罪 / reference_entities / 时长比例 / BGM lint 等工程坑。

## ❌ 反例（影评式 / 是非式 / 复读式，全部踩过）

> 反例 1（影评式）：「这段视频是否符合科幻武侠片的诉求？整体节奏和内容有什么优缺点？」
> → VLM 输出"叙事支撑不足、武侠属性偏弱"这种**不可执行**的影评。

> 反例 2（是非式 · 13:06 那次踩的坑）：「请检查这段视频内容是否符合要求：橘色短毛小猫戴白色领结...回答是/否，以及不符合的地方」
> → VLM 回"是，视频内容完全符合要求。"——agent 拿这个当过审了，但根本没让 VLM 做导演式判断。
> → 真正问题（如"领结颜色对吗 / 音箱位置对吗 / 动作连贯性"）一个都没问到。

> 反例 3（复读式 · 15:02 小猪跳舞踩的坑）：「请审核这张场景图：1.是否符合描述：晴朗白天的绿色草坪... 2.整体是否无违和、无穿模？」
> → VLM 回"整体结论：过审。该场景完全符合描述要求，不存在违和、穿模问题，无需修正。"
> → 这是 QA 验收报告不是导演审片。**没有任何审美判断**——光的质感、花的密度、构图的呼吸感全没提。
> → 根因：agent 把"期望要点"写成了是非题清单，VLM 就照清单复读"是/无问题"。

> 反例 4（结论顺序错位 · 16:32 彩虹小马踩的坑）：question 末尾写「**最后明确给出整体结论：过审/不过审**」
> → VLM 照办把结论放最后："### 核对检查结果：1.... 2.... ### 整体结论：过审"
> → agent 用 `answer.startswith('整体结论：过审')` 解析直接失败、误判通过 → 跑下一步带病前进。
> → 根因：agent 写"最后给出"，跟 md 正例"**第一行**：整体结论..."冲突。
> → 修复：question 里**禁止**出现"最后/末尾给出结论"，必须明确写「**第一行（必须从这一句开头）**：整体结论：过审 或 整体结论：不过审」。

> 反例 5（基准图过审后又加新约束 · 16:32 同项目）：front 已过审且实际是"双足拟人小马"，
> agent 在生成 back 时强加 prompt"**四足卡通可爱小马，四足肢体结构正确**" → 跟工具自动注入的 front ref_image_url 打架 → 模型混乱 → 不过审 → 重做 → 又不过 → ask_user。
> → 根因：agent **没把基准图视为已确立的造型基准**，还想在后续视图里"修正"它。
> → 修复：front 一旦过审，造型就锁死。后续 side/back 的 prompt 必须**继承 front 的所有特征**，不许加新约束。如果觉得 front 不对，**先重做 front**，再开其他视图。

**修复**：用「过审/不过审」当终极判断词；期望要点写成**正向描述**而非是非题；强制 VLM 在过审时也要说出 2-3 句**它自己的导演观感**，不许复读。

## ✅ 正例（三段式 question · 按产物类型直接抄）

**结构 = 三段**，**顺序不可调**：身份代入 + 整片 context → 导演发散思考维度 → 末尾格式约束。

> ⚠️ **核心原则**：question 不是"我要 X，你看是不是 X"的对答案题，而是给 VLM **整片 context + 导演发散视角**，让它**自己**用专业眼光从多维度评判产物配不配上这部片。
>
> **第二段绝对禁止写"我要的样子是 A+B+C+D"清单**——清单只会让 VLM 照清单复读"是的我看到了 A+B+C+D"，发现不了"虽然元素都对，但整体味道不对"。要写成"作为 XX 类型片导演，你会从哪些维度去看这个产物"——给思考框架，不给答题模板。

---

### 场景 A · entity 基准图（character 三视图 / scene 宽景 / prop 默认视图）

**导演关心什么**：这个 entity **立不立得住**？造型/质感/风格是不是这部片用得上的？后面 N 个 shot 都要用它，不立得住后患无穷。

```
─── 第一段 · 身份代入 + 整片 context ─────────────────────────────────
你是本片导演 A 类型片（如"30 秒公益短视频，主题校园霸凌警示"/"武侠动作短片"
/"萌宠生活记录"）作品的总导演。本片的整体调性是 <暖色温情 / 冷峻克制 /
夸张幽默...>，目标观众是 <家长群体 / 短视频平台年轻用户 / ...>。

现在你正在审核 <角色名/场景名/道具名> 的<基准三视图之一/场景宽景/道具默认图>，
这是 entity-first 流程的第一步——**这张图一旦定稿，后续整片所有 shot 都会
拿这张图当参考**。如果这张图本身造型/风格不立得住，后面 5 个 shot 全要返工。

你本人是有审美 taste 的导演，不是只会勾选条件清单的 QA。

─── 第二段 · 导演发散思考维度（给框架，不给答案）──────────────────
作为这部片的导演，你需要从下面几个维度评判这张图配不配做基准：

1. **造型本身够不够"立得住"**：人物/场景/物体的视觉气质，是不是有"主角感"
   /"这个片子专属"的辨识度？还是太通用太套图？
2. **风格质感跟整片调性兼不兼容**：色温 / 材质 / 光感 / 笔触跟你脑子里这部
   片的调性是同一套审美吗？放进 storyboard 其他镜头里会不会突兀？
3. **细节有没有硬伤**：解剖结构 / 比例 / 透视 / 物理逻辑（影子方向、材质反光、
   多余穿模等）— 不是要完美，是不能有让观众一眼出戏的硬伤。
4. **<场景类专属>** 留白 / 构图呼吸感 / 后续放主角的位置 — 场景图不是风景
   壁纸，是给主角"住进去"的舞台，得留得出戏剧空间。
   **<character 类专属>** 三视图之间是否同一个角色？造型一致性？
   **<prop 类专属>** 道具是这个角色/场景里"用得起"的吗？质感符合身份吗？

不用每条都答，**挑你作为导演真正在意的、能直接影响后续出片的点说**。

─── 第三段 · 输出格式约束（末尾，不可前置）────────────────────────
请你**独立观察**这张图实际拍出来什么样，**不要被上面的诉求/调性描述带偏**——
你看到什么就说什么。然后凭你脑子里那部成片的样子判断这张基准图配不配立稿。

严格按下面的格式输出：

  1) **第一行（必须从这一句开头）**：「整体结论：过审」 或 「整体结论：不过审」
     二选一。禁止"含糊 / 基本 / 大致 / 待定 / 部分符合"等中间态。

  2) 第二段（**至少 3-5 句，发散性导演评语**）：
     · **过审**：说出你作为导演具体看上了哪些点（光感 / 质感 / 留白 / 主角感
       / 风格命中度...）— 任一个真正打动你的点。**禁止**写"完全符合描述
       / 无违和"这种 QA 验收语。
     · **不过审**：说出**导演视角的问题诊断 + 你想要的样子**——
       具体说哪里不对（造型/质感/光感/构图/硬伤...）、要改成什么样子
       （从导演审美出发，描述目标观感而非工程参数）。
       **不需要写成 prompt 格式 / 不需要用特殊符号 / 不需要列 reference_entities**——
       这些是 agent 拿你的诊断走 PE 7 步翻译时该做的事，你只管"导演说话"。

不打分、不分维度、不写小标题。
```

### 三个场景的核心差异（一图明白）

| 场景 | 导演关注重心 | 不过审 → agent 走 PE 翻译后喂回 |
|---|---|---|
| A · entity 基准图 | 造型立不立得住 / 风格命中度 / 细节硬伤 | `gen_image` / `entity_add_view` |
| B · 单 shot 视频 | 叙事意图 / 演员表演 / 段间承接 | `gen_video_t2v` |
| C · 全片成片 | 情绪曲线 / BGM 贴合 / 完成度 | `gen_audio_bgm`（BGM 层）或 `gen_video_t2v`（画面层）|

---

### 共同铁律（三个场景都适用）

1. **第一段必须给"整片 context"**：本片是什么类型 / 调性 / 节奏 / 在流水线哪一步——VLM 没有这个锚点就只会照清单复读
2. **第二段绝对不写"我要的样子是 A+B+C+D"清单**：写"作为 XX 导演你会看哪些维度"——给思考框架不给答题模板
3. **第三段格式约束必须放末尾**：position 错了直接踩死罪 4
4. **不过审要的是"导演诊断+想要的样子"，不是工程化 prompt**：VLM 只管说"哪里不对+想要什么"，agent 走 PE 7 步翻译成合规 prompt 再喂回工具

---

> **三段为什么不能调顺序？**
> VLM 按 token 顺序处理 prompt，**第一句话定调整段提问的框架**：
> - 第一段开头先身份代入 + 整片 context → VLM 切到导演视角，知道在评判什么作品
> - 第二段中段才能放发散思考维度（VLM 已经"是导演了"，不会照清单复读）
> - 第三段最后才接格式约束（这是给 agent 解析用的"输出契约"，不是给 VLM 当提问的）
>
> 把第三段塞最前面 = VLM 第一眼看到"过审/不过审"，整段提问被框成是非题，谄媚 100% 命中（已踩坑见文档顶部 4 大死罪表 · 死罪 4 案例）。
>
> **关键点**：让 VLM 出**导演视角的"问题诊断+想要的样子"**，不是出**成品 prompt**。agent 拿到不过审 → 把 VLM 的诊断当"创作意图原文" → **必须 file_read [skill_prompt_engineering.md](../skill_prompt_engineering/skill_prompt_engineering.md) 走 PE 7 步 checklist 翻译** → 翻译成符合规范的 prompt（4 类符号 / 5 大死罪 / reference_entities / 时长比例 / BGM lint 全过）→ 再喂回 `gen_image / gen_video_t2v / gen_audio_bgm`。
>
> **VLM ≠ PE 工程师**——这是两个不同的专业，不能混。VLM 只懂导演审美，agent 才懂 Seedance/Seedream/BigMusic 的工程规范。

## 链式片子专用必查项

链式段视频出来后，除了通用过审判断，**必须问**：

- "这一段的开头画面，是否自然承接了上一段的结尾？人物/场景/动作是否一致？"
- "和上一段对比，主体的造型/颜色/视角是否保持一致？"
- "这一段的对白音色 / 现场音效，和上一段是否一致或合理过渡？"
  > ⚠️ **不要问 BGM**——本项目 Seedance 不出 BGM，整片 BGM 是阶段 5 通过 `gen_audio_bgm` 一次性铺底，不会有"段间配乐"问题。

发现衔接断裂 → 重做（`reference_video_url` 仍指上一段视频，prompt 强调"承接"）。

## VLM 模式选择

`vlm_understand(mode=...)`：
- `auto`（默认）：视频走原生理解（不抽帧），图片走多图理解
- `frames`：强制抽帧。视频很长想省钱时用
- `video`：强制走视频原生理解

视觉评估**只有这一个工具**。`extract_frames` 是不调 VLM 的纯抽帧，debug 看视频内容时才用。
