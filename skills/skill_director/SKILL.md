# Skill · 编导转译（从剧本到可拍方案）

> **何时读我**：剧本已经定稿、准备进入素材/分镜/视频生成之前；或者发现“剧本很好，但成片平、乱、抓不住人、不会剪”的时候。  
> **我不负责**重新写完整剧本，也不直接调生成工具；我负责把剧本翻译成导演和剪辑师能执行的拍摄方案。

## 核心定位
剧本是“发生了什么”，编导转译是“观众先看到什么、怎么被钩住、每个信息用什么镜头交代、最后怎么剪成一条顺的片”。  
如果跳过这一层，模型通常会把好剧本拍成“人在场景里做动作”的流水账。

## 输入与输出
输入必须是已经过剧本 skill 审查的定稿剧本，至少包含人物、场景、剧情、逐字台词、关键道具和统一风格方向。

输出到工作目录，建议命名为 `director_plan.json` 或写入结构化 `storyboard.json`，至少包含：
- `opening_hook_plan`：黄金三秒方案。
- `narrative_order`：最终成片叙事顺序，可线性、插叙、倒叙、结果前置。
- `must_keep_global`：全片不可丢失信息。
- `shots[]`：逐片段编导拍摄表。
- `editing_decisions[]`：剪辑决策表。
- `audio_plan`：整片音频处理计划。

## 一、先重排叙事，不要被剧本时间顺序绑死
剧本可以按事件发生顺序写，但成片未必按这个顺序拍。先判断哪种结构最抓人：
- **线性推进**：适合动作因果强、冲突一开始就很明确的内容。
- **结果前置**：先给崩溃、危险、告别、反转结果，再切回“几分钟前”。适合悬疑、遗憾爱情、反转短片。
- **插叙**：用回忆、证据、旧物、屏幕内容穿插解释情感或真相。适合爱情、悬疑、人物秘密。
- **倒叙**：从结局或强情绪点开始，逐步揭开原因。适合强反转、强遗憾、强悬念。

必须写清：为什么选择这种叙事顺序，它如何服务前三秒钩子和最终情绪落点。

## 二、黄金三秒方案
第一个片段必须承载黄金三秒，不允许用平淡铺垫开场。黄金三秒至少使用一种钩子：
- **视觉钩子**：异常画面、危险动作、强反差、崩溃表情、不可思议的屏幕/纸条/物件。
- **信息钩子**：一句反常台词、可读文字、倒计时、死亡/分手/背叛/真相提示。
- **声音钩子**：门铃、尖锐提示音、电话录音、喘息、键盘声、突然静音。
- **结构钩子**：结果前置、插叙切入、先给结局再回到开端。

输出格式：
```json
"opening_hook_plan": {
  "hook_type": "结果前置/异常信息/强冲突/声音钩子",
  "first_3_seconds": "观众前三秒具体看到和听到什么",
  "why_it_hooks": "为什么能让观众继续看",
  "return_point": "钩子之后如何回到主线"
}
```

## 三、不可丢失信息清单
从剧本中抽出所有不能丢的信息。任何一项进入后续视频 prompt 前都不能被概括。

必须抽取：
- 逐字对白。
- 屏幕/纸条/手机/合同/日历/聊天记录上的逐字文字。
- 时间、日期、数字、倒计时。
- 推动剧情的道具状态。
- 人物动作破绽。
- 反转句、告别句、情绪爆点。
- 角色声音描述。
- 关键环境声。

输出格式：
```json
"must_keep_global": {
  "dialogue": [],
  "screen_text": [],
  "key_visual_info": [],
  "sound_cues": [],
  "voice_profiles": []
}
```

## 四、关键信息镜头化
凡是推动剧情的信息，都必须安排成观众能看见、听见、理解的镜头。禁止只写“他发现异常”“她看到消息”“两人争吵”。

转换规则：
- 文字信息 → 屏幕/纸条/手机/合同大特写，文字清晰可读，并逐字写出内容。
- 时间数字 → 时钟/手机/电脑角落特写，写出具体显示。
- 道具线索 → 道具插入镜头 + 人物反应镜头。
- 情绪变化 → 面部近景、手部动作、停顿、呼吸、沉默。
- 反转信息 → 先铺垫，再给可读/可听的明确揭示，再给反应。

输出格式：
```json
"key_info_shots": [
  {
    "info": "要传达的关键信息",
    "shot": "用什么镜头传达",
    "must_be_visible": true,
    "must_be_readable": true
  }
]
```

## 五、导演镜头节拍表
每个片段必须写导演镜头节拍表，不允许只写剧情摘要。每个 beat 是一个可拍的镜头节拍。

输出格式：
```json
"director_beats": [
  {
    "beat": 1,
    "camera": "景别/机位/运镜",
    "action": "画面动作",
    "visible_text": "如有文字，逐字写出",
    "dialogue": {"speaker": "角色", "text": "逐字台词", "voice": "完整音色"},
    "sound": "环境音/动作声",
    "purpose": "这个镜头给观众什么信息或情绪",
    "references": ["需要的角色/场景/道具/关键帧"]
  }
]
```

检查标准：
- 每个 beat 都能被画出来。
- 每个 `must_keep` 都能追溯到某个 beat。
- 一个片段装不下太多 beat 时，拆成多个片段。
- 有对白就保留逐字台词和完整音色；无对白就不要写口型语义词。

## 六、剪辑决策表
视频片段生成前就要预设剪辑逻辑，合成时再按实际成片微调。不要到最后只把片段硬拼。

每个衔接点必须说明：
```json
"editing_decisions": [
  {
    "from": "s01",
    "to": "s02",
    "transition": "cut / crossfade / fade_to_black / flash / match_cut",
    "duration": 0.8,
    "reason": "为什么这样剪",
    "audio_rule": "保留原声/去原声/承接环境声/BGM延续"
  }
]
```

选择原则：
- 同一场景连续动作、惊吓、情绪突变：可以硬切。
- 时间跳跃、回忆、空间转换、情绪渐变：优先考虑叠化、黑场或闪白。
- 强反转揭示：前后镜头要有反应承接，不能突然断。
- 抖音情绪短片：少量转场即可，重点是节奏和情绪连续，不要炫技。

`compose_timeline` 可以作为这份计划的结构名，但它是编导计划，不要求工具层有同名工具。工具层只负责执行已决定的 `concat / crossfade / fade / audio_process` 等原子动作。

## 七、音频处理计划
音频必须先设计，再处理。禁止一段有 Seedance 原声、一段静音、一段随机 BGM，最后硬混。

输出格式：
```json
"audio_plan": {
  "dialogue_strategy": "Seedance原生对白 / 后期TTS / 无对白",
  "keep_original_audio": ["s01"],
  "strip_original_audio": ["s02", "s03"],
  "bgm": {
    "use_unified_bgm": true,
    "segments": [],
    "volume": 0.18,
    "fit_to_final_duration": true,
    "fade_in": 0.5,
    "fade_out": 1.2
  },
  "audio_process_steps": [
    {"action": "strip", "target": "需要去原声的视频"},
    {"action": "fit_duration", "target": "BGM适配成片时长"},
    {"action": "amix", "target": "最终混音"}
  ]
}
```

## 八、编导方案交付
完成编导转译后，输出可被后续素材、分镜、视频生成、审查、合成或其他制作流程消费的执行方案。交付时必须明确：
- 资产清单来自哪些 shot，未使用资产删掉。
- 每个 shot 的参考素材列表。
- 每个 shot 的导演节拍和不可丢失信息。
- 剪辑决策表和音频处理计划。

如果后续 prompt、分镜、合成结果没有逐项继承本方案输出的内容，视为编导转译失败，必须回到这一层重做。
